## Getting Started

1. **Fork the repository** on GitHub
2. **Clone your fork** locally: `git clone https://github.com/YOUR_USERNAME/claude-forge.git`
3. **Add upstream remote**: `git remote add upstream https://github.com/rynhardt81/claude-forge.git`
4. **Create a feature branch**: `git checkout -b feature/your-feature-name`

## Making Changes

### Understanding Claude Forge Architecture

Before making changes, familiarize yourself with:

- **Skills**: Located in `.claude/skills/`, these define workflows that Claude can execute
- **Agents**: Located in `.claude/agents/`, these define specialized personas for different roles
- **Templates**: Located in `.claude/templates/`, these define document structures
- **Hooks**: Located in `.claude/hooks/`, these enforce security and validation rules

### Types of Contributions

#### 1. Feature Additions

- New skills or workflow improvements

#### 2. Bug Fixes

- Include links to any relevant documentation

#### 3. Documentation

- README improvements
- Setup guide clarifications
- Example projects
- Tutorial content

#### 4. Configuration

- Template improvements
- Configuration preset adjustments
- Hook enhancements

## Commit Guidelines

We follow conventional commit format:

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Commit Types

- **feat**: A new feature or skill
- **fix**: Bug fixes
- **docs**: Documentation changes
- **refactor**: Code/structure reorganization (no feature changes)
- **test**: Test additions or improvements
- **chore**: Build process, dependencies, tooling
- **style**: Formatting, linting changes
- **perf**: Performance improvements

### Commit Examples

```
feat(dispatch): Add intent detection for brainstorming skill
fix(hooks): Resolve gate-check validation errors for .env files
docs: Add project memory system documentation
chore: Update migration scripts for v1.4.0
```

### Scope Examples

- `dispatch`: Intelligent dispatch system
- `hooks`: Claude Code enforcement
- `agents`: Agent personas
- `skills`: Workflow skills
- `templates`: Document templates
- `memory`: Project memory system
- `docs`: Documentation

## Pull Request Process

1. **Before submitting** a PR, ensure:
   - You've synced with the latest `main` branch: `git pull upstream main`
   - Your changes follow the code standards
   - You've tested your changes thoroughly
   - Documentation is updated if needed

2. **Create a descriptive PR title** following commit guidelines:
   - ✅ `feat(dispatch): Implement parallel agent scheduling`
   - ❌ `updates` or `fixes stuff`

3. **Fill out the PR description template** with:
   - What problem does this solve?
   - What changes were made?
   - How to test the changes?
   - Any related issues (use `Closes #123`)

4. **Wait for review** - maintainers will provide feedback

5. **Address feedback** - make requested changes and push updates

6. **Merge** - once approved, your PR will be merged!

## Code Standards

### General Guidelines

- **Clarity first**: Code should be readable and self-documenting
- **Follow existing patterns**: Match the style of surrounding code
- **Keep it simple**: Avoid unnecessary complexity
- **Document complex logic**: Add comments explaining the "why", not the "what"

### Markdown Standards

- Use proper heading hierarchy (H1 → H2 → H3)
- Include a table of contents for long documents
- Use code blocks with language specification: ` ```bash `
- Link to related documentation

### Shell Script Standards

- Add comments explaining non-obvious sections
- Follow existing indentation (usually 2-4 spaces)
- Include error handling
- Add usage examples in comments

### Python Standards (if applicable)

- Follow PEP 8 style guide
- Include docstrings for functions
- Use type hints where helpful
- Add unit tests for new functions

## Testing

### Manual Testing

1. Create a test project: `mkdir test-project && cd test-project`
2. Clone your changes: `git clone <your-fork> .claude`
3. Test with Claude Code: `claude`
4. Run `/new-project "Test Project"` to validate setup
5. Test relevant skills: `/reflect status`, `/implement-features --help`, etc.

### Documentation Testing

- Verify all links are correct
- Test all code examples
- Ensure screenshots are up-to-date (if any)
